③ HA / DR パターン
  • マルチAZ → 同一リージョン内の高可用性（RDS Multi-AZ、ALB、EFS）
  • マルチリージョン → 地理的DR（Route 53 フェイルオーバー、S3 CRR、Aurora Global）
  • Auto Scaling + ELB → スケーラビリティ + 自動復旧
  • RTO / RPO の使い分け
    • Backup & Restore: RPO/RTO 長い・最安
    • Pilot Light: コア機能のみ常時稼働
    • Warm Standby: 縮退版を常時稼働
    • Multi-Site Active/Active: RTO≈0・最高コスト
④ セキュリティの考え方
  • 最小権限の原則 → IAM ポリシーは必要最小限のアクションだけ許可
  • 暗号化 → 保存時: SSE-S3 / SSE-KMS / SSE-C、転送時: TLS / HTTPS
  • ネットワーク分離 → Public/Private サブネット、NAT Gateway、Security Group vs NACL
  • モニタリング → CloudTrail（API記録）/ GuardDuty（脅威検知）/ Config（設定変更監視）
  • WAF vs Shield → WAF: L7 (SQLi/XSS)、Shield Standard: DDoS 自動、Shield Advanced: 手動対応付き
⑤ コスト最適化のキーワード
  • Reserved Instances / Savings Plans → 1〜3年コミット、最大72%割引
  • Spot Instances → 中断許容バッチ処理に最安
  • S3 ライフサイクル → Standard → IA → Glacier で段階的に安く
  • Lambda / Fargate → アイドル時コスト0のサーバーレス
  • コスト可視化 → Cost Explorer、Budgets アラート
⑥ 問題を読むときの注意点
  • 「最もコスト効率が良い」→ サーバーレス / Spot / Reserved を優先
  • 「最小限の運用負荷」→ マネージドサービス（RDS > EC2上のMySQL）
  • 「既存オンプレと接続」→ Direct Connect / VPN / Storage Gateway
  • 「リアルタイム処理」→ Kinesis / Lambda / SQS FIFO
  • 「疎結合」→ SQS でキュー分離、SNS でファンアウト
